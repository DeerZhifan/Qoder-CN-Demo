所有涉及代码生成、文件创建、架构变更的任务，必须严格遵循以下六阶段闭环流程。禁止跳过任何阶段。

流程概览：**探索 → 规划 → 确认 → 执行 → 检查 → 反馈**

---

## 一、探索阶段

收到任务后，先理解现状再做决策。主动阅读相关代码和配置，建立对当前项目状态的准确认知。

### 探索内容

1. **阅读相关文件**：打开任务涉及的现有代码、配置文件、POM 依赖，理解当前实现
2. **理清上下文**：确认项目的模块结构、命名规范、技术栈版本、已有的类和方法
3. **识别约束**：检查是否存在框架约定（如 cdp-backend-rule）、编码规范、依赖限制

### 探索阶段的约束

- 只读不写，不修改任何文件
- 不确定的信息必须向用户提问，禁止基于假设继续
- 探索结论作为规划阶段的输入依据，不单独输出给用户

---

## 二、规划阶段

基于探索结果，输出一份结构化的执行计划。

### 计划内容

1. **任务理解**：用一句话复述你对任务目标的理解
2. **影响范围**：列出需要新建、修改、删除的文件清单（精确到文件路径）
3. **执行步骤**：按顺序列出每一步的具体操作，标注步骤编号
4. **检查策略**：说明执行完成后将如何验证正确性（编译检查 / 语法检查 / 逻辑审查）
5. **依赖与风险**：指出前置条件、潜在冲突或需注意的点

格式示例：

```
## 执行计划

**任务理解**：在 CDP 项目中新增订单管理业务模块

**影响范围**：
- 新建：leatop-cdp-myapp-order/（5 个子模块，共 15 个文件）
- 修改：pom.xml（根 POM 新增 module 声明）
- 修改：leatop-cdp-myapp-main/pom.xml（新增 boot-starter 依赖）

**执行步骤**：
1. 创建业务模块聚合 POM
2. 创建 api 子模块（Business 接口、DTO、QO）
3. 创建 service 子模块（PO、DAO、Service、BusinessImpl）
4. 创建 controller 子模块
5. 创建 boot-starter 子模块（自动配置类 + imports 注册）
6. 创建 cloud-starter 子模块（自动配置类 + imports 注册）
7. 注册到根 POM 和 main 模块

**检查策略**：
- 检查所有 POM 的 parent/artifactId/依赖引用是否正确
- 检查 Java 文件的 package 声明与目录结构是否一致
- 检查 AutoConfiguration.imports 注册路径是否正确

**依赖与风险**：
- 需确认模块名称、实体名、表名等参数
```

### 规划阶段的约束

- 不确定的参数必须向用户提问，禁止猜测或使用默认值
- 如果任务涉及多个独立子任务，分别列出，由用户决定执行顺序或范围
- 规划阶段禁止创建或修改任何文件

---

## 三、确认阶段

将执行计划完整展示给用户，等待明确确认后再执行。

### 确认阶段的约束

- 必须以明确的提问结束，例如："以上计划是否可以开始执行？如需调整请说明。"
- 用户回复"确认"、"可以"、"开始"等肯定词后方可进入执行阶段
- 用户提出修改意见时，更新计划并重新确认，不可带着未确认的修改直接执行
- 如果用户明确表示跳过确认（如"直接做"），可以省略本阶段

---

## 四、执行阶段

按照已确认的计划逐步执行，每完成一个步骤进行简要播报。

### 播报格式

```
[步骤 3/7] 创建 service 子模块 ✓
  - 新建 leatop-cdp-myapp-order-service/pom.xml
  - 新建 OrderPO.java, OrderDAO.java
  - 新建 OrderService.java, OrderServiceImpl.java
  - 新建 OrderBusinessImpl.java
```

### 执行阶段的约束

- 严格按照确认的计划和顺序执行，不可自行增减步骤
- 遇到计划外的问题（如文件冲突、依赖缺失），立即停下并向用户说明情况和建议方案，等待指示后再继续
- 每个步骤执行完毕后再执行下一步，不可一次性输出所有结果
- 生成的代码必须遵循项目现有的编码规范和框架约定（参照 cdp-backend-rule）

---

## 五、检查阶段

执行完成后，主动对所有变更进行自检。如果发现问题，立即修正，不需要等待用户指示。

### 检查清单

#### POM 文件检查
- parent 的 groupId、artifactId、version 是否正确
- 当前模块的 artifactId 是否符合命名规范
- 依赖引用的 artifactId 是否与目标模块一致
- 子模块间引用是否省略了 version（由 dependencyManagement 管理）
- modules 列表是否完整

#### Java 文件检查
- package 声明是否与文件所在目录路径一致
- import 语句中的包名是否正确（特别注意模块间的引用）
- 类名是否与文件名一致
- 继承/实现关系是否正确（如 BasePo、BaseMapper、IService、ServiceImpl）
- 注解参数是否正确（如 @TableName、@MapperScan、@ComponentScan、@FeignClient）

#### 配置文件检查
- YAML 缩进是否正确（使用空格，不使用 Tab）
- 占位符变量名是否前后一致
- AutoConfiguration.imports 中的全限定类名是否与实际类路径一致

#### 一致性检查
- 所有文件中的模块名称、包名、类名引用是否统一（无拼写不一致）
- 新模块是否已注册到根 POM 的 modules 中
- main 模块是否已添加对应的 boot-starter 依赖

### 自修正机制

- 检查发现问题时，直接修正，修正后重新检查该项
- 每轮修正后输出修正说明：

```
[自检修正] 发现 2 个问题，已自动修正：
  1. leatop-cdp-myapp-order-service/pom.xml 中 api 依赖的 artifactId 拼写错误 → 已修正
  2. OrderServiceImpl.java 的 package 声明与目录不一致 → 已修正
```

- 最多执行 3 轮修正。如果 3 轮后仍有问题无法解决，停止修正，在反馈阶段向用户说明

### 检查阶段的约束

- 检查是必须执行的，不可跳过
- 修正范围仅限于本次任务产生的变更文件，不可修改任务范围外的代码
- 修正后必须重新验证，确保修正本身没有引入新问题

---

## 六、反馈阶段

检查通过后，输出一份结构化的执行报告。

### 报告内容

1. **执行结果**：逐步骤标注完成状态
2. **自检结果**：检查是否全部通过，如有修正需列出修正内容
3. **文件清单**：列出所有新建和修改的文件路径
4. **后续操作**：用户接下来需要手动完成的事项

格式示例：

```
## 执行报告

**执行结果**：7/7 步骤完成

| # | 步骤 | 状态 |
|---|------|------|
| 1 | 创建业务模块聚合 POM | 完成 |
| 2 | 创建 api 子模块 | 完成 |
| ... | ... | ... |

**自检结果**：通过（第 1 轮发现 1 个问题，已自动修正）

**新建文件**（15 个）：
- leatop-cdp-myapp-order/pom.xml
- leatop-cdp-myapp-order/leatop-cdp-myapp-order-api/pom.xml
- ...

**修改文件**（2 个）：
- pom.xml（新增 module 声明）
- leatop-cdp-myapp-main/pom.xml（新增 boot-starter 依赖）

**后续操作**：
1. 补充 OrderDTO / OrderPO 中的业务字段
2. 创建 Flyway 建表脚本
3. 执行 `mvn clean install -DskipTests=true` 验证编译
```

## 七、特殊场景处理

### 简单任务

对于影响范围小于 3 个文件的简单修改（如修改一个配置值、修复一个明显的 bug），可以合并探索、规划、确认阶段为一句话说明，但检查阶段不可省略：

```
这是一个小改动：将 application-dev.yaml 中的数据库端口从 3306 改为 3307。直接修改。
[自检] YAML 格式正确，变更符合预期 ✓
```

### 用户中途变更需求

- 立即停止当前执行
- 基于已完成的步骤和新需求，重新进入探索阶段评估影响
- 输出更新后的计划，重新进入确认阶段

### 执行出错

- 说明错误原因和影响范围
- 提供修复方案（回滚 / 修正 / 跳过）
- 等待用户选择后再继续
